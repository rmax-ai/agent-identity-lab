## ADR-002: JWT Session Credentials

Short-lived JWTs were chosen as the session credential because they are easy for multiple services to validate and carry enough signed context to support stateless verification in the common path. A session token can include agent identity, acting user, scopes, trace information, and expiry without requiring every consumer to query the database on each request.

The main alternative was an opaque session identifier with mandatory introspection. That would reduce claim exposure but increase coupling and latency between services. For this repo, the better tradeoff is a signed token paired with server-side revocation checks where stronger control is needed.

This decision works only because tokens are short-lived and distinct from downstream credentials. The JWT proves approved session context; it does not replace revocation checks, policy evaluation, or tool-specific credential leasing.

## ADR-005: OPA Policy Adapter

The policy layer is abstracted behind an adapter so the repo can support both a Python-native evaluator and an OPA-backed implementation. This lets the MVP remain easy to run locally while preserving a path to more formal, centrally managed policy logic in production deployments.

Choosing OPA as the primary external policy engine keeps policy definitions declarative and testable. It also avoids baking every rule into API handler code. The adapter boundary matters because it prevents the rest of the system from depending on one specific policy execution engine.

The Python adapter remains useful as a fallback and for local tests, but the design intent is that policy evaluation stays a replaceable subsystem with a stable input and output contract.
